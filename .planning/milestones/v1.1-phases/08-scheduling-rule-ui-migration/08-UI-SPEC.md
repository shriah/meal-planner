## Color

All colors from shadcn CSS variables (olive theme). Do not hardcode hex values; use CSS variable references.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` (oklch 1 0 0 = white) | Page background, form surface |
| Secondary (30%) | `--muted` / `--card` | Fieldset areas, tag filter grid background, preset cards |
| Accent (10%) | `--primary` (oklch 0.852 0.199 91.936 = golden-olive) | Active toggle switch, Save Rule button (primary variant), active TabsTrigger |
| Destructive | `--destructive` (oklch 0.577 0.245 27.325 = red) | Delete confirmation alert, Delete button variant |

Accent reserved for:
- Save Rule button (primary Button variant) — the only primary CTA in the form
- Rule enabled/disabled toggle switch (`bg-primary` when enabled)
- Active `TabsTrigger` selected state (shadcn default behaviour)

Muted foreground (`--muted-foreground`): impact preview descriptions, field helper text, preset card secondary text.

Warning color: `border-amber-500 bg-amber-50 text-amber-900` — impact preview zero-match alert. Inherited from `RuleImpactPreview`. Keep exact class strings; do not replace with design tokens.

## Component Inventory

New component to build: `SchedulingRuleFields.tsx`

### SchedulingRuleFields layout (top to bottom)

```
[Effect selector — Tabs]
  TabsList: "Only allow" | "Always include" | "Never include"
  (maps to: filter-pool | require-one | exclude)

[Match mode — RadioGroup]  (appears after effect is selected)
  ● By tag   ○ Specific component

[Tag fields — 2-col grid]  (shown when match mode = "By tag")
  Dietary | Protein
  Regional | Occasion
  (reuse exact Select pattern from DayFilterFields)

[Component picker — two-step]  (shown when match mode = "Specific component")
  Step 1: Select type (Base / Curry / Subzi) — shadcn Select
  Step 2: Combobox filtered to selected type — shadcn Combobox

[Scope — Days]  (optional, appears below match config)
  Checkbox group — reuse DayFilterFields day-checkbox pattern exactly

[Scope — Slots]  (optional)
  Checkbox group — reuse DayFilterFields slot-checkbox pattern exactly
```

### Effect tab labels (plain English — per D-01)

| Tab label | Internal value | Plain description shown to user |
|-----------|---------------|----------------------------------|
| Only allow | filter-pool | Only components matching this rule are eligible |
| Always include | require-one | At least one matching component is always picked |
| Never include | exclude | Matching components are removed from the pool |

### Match mode radio (per D-02)

Rendered as two radio inputs with labels:
- `● By tag` — reveals the 2×2 tag filter grid (Dietary / Protein / Regional / Occasion)
- `○ Specific component` — reveals the two-step component picker

Use shadcn `RadioGroup` + `RadioGroupItem` (not yet installed — see Registry Safety section).

Fallback if RadioGroup not added: use two `<button>` elements with `role="radio"` and `aria-checked`, styled with `border` + `bg-primary/10` for selected state. Prefer installing RadioGroup.

### Component picker (two-step — per D-03)

Step 1: `Select` — options: Base / Curry / Subzi (no extras).
Step 2: `Combobox` — filtered to components matching selected type; shows component name; value is numeric `component_id`.

When type not yet selected, Step 2 combobox is disabled with `placeholder="Select type first"`.

## Form Validation Contract

Save Rule button (`Button type="submit"`) is **disabled** until all of:
1. `name.trim() !== ''`
2. `effect !== ''` (one of the three tabs is selected)
3. Match mode is selected (`match.mode !== ''`)
4. EITHER: `match.mode === 'tag'` AND at least one tag filter key has a non-undefined value (dietary_tag OR protein_tag OR regional_tag OR occasion_tag)
5. OR: `match.mode === 'component'` AND `component_id !== null`

Days and slots are optional — no minimum required (null = all days / all slots per Phase 7 D-09).

Source: CONTEXT.md `## Claude's Discretion`, CONTEXT.md `<specifics>`

## Impact Preview Contract

`RuleImpactPreview` new case for `scheduling-rule`:

**Tag mode:** Count components matching ALL non-empty tag filter keys (same logic as current `day-filter` case). Display: `"This rule affects {N} of {total} components."` Show amber warning alert if N === 0.

**Component mode:** Show component name (look up by `component_id`). Display: `"This rule applies to {componentName}."` If `component_id` is null, show nothing (impact is null).

Both cases: only render when `effect !== ''` AND `match.mode !== ''` AND the relevant fields are populated enough to compute a meaningful count.

## Interaction States

### Rule type tab set (after Phase 8)

Before any tab selected: Save button disabled.

Tab set: `no-repeat` | `scheduling-rule`

Removed tabs: `day-filter`, `require-component` (per D-05).

### Effect selector (inside SchedulingRuleFields)

Three tabs: "Only allow" | "Always include" | "Never include"

Selecting a tab sets `effect`. Does NOT reset `match` — preserves match config if user switches effects.

### Match mode radio

Visible only after `effect !== ''`.

Toggling between "By tag" and "Specific component" resets the match sub-fields (dispatches `SET_MATCH_MODE` which resets filter or component_id per existing reducer).

### Tag filter selects

Each Select shows "Any" as the unset option (value `'any'` maps to undefined in filter). Selecting "Any" removes the key from the filter object.

### Component picker two-step

Type Select → enables Combobox. Changing type resets `component_id` to null (handled in reducer or local state in SchedulingRuleFields).

### Preset cards (RuleEmptyState)

Three scheduling-rule presets + one no-repeat preset. Cards navigate to `/rules/new?preset={id}`. Styling: `p-4 rounded-lg border border-border opacity-50 hover:opacity-80 cursor-pointer transition-opacity` — unchanged from current implementation.

### Rule row toggle

Custom toggle switch (inline `<button role="switch">`) — NOT a shadcn Switch component. Existing pattern in RuleRow; unchanged.

### Delete confirmation

Inline expand below the rule row (no modal/dialog). Alert variant destructive. Two buttons: Delete (destructive) + Keep rule (ghost). Pattern unchanged from existing RuleRow except cancel label updated to "Keep rule".

## Migration UI Contract

Migration is **silent** — no banner, no notification, no UI feedback (per D-06).

Migrated rules appear in the rules list immediately after Dexie v3 upgrade runs at app startup. `ruleDescriptions.ts` already handles `scheduling-rule` descriptions — list rendering requires no change.
