## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, checkbox-to-label gap (`gap-1`) |
| sm | 8px | Compact element spacing, button gaps, checkbox-to-label in day/slot groups (`gap-2`) |
| md | 16px | Default element spacing, form field stack (`space-y-4`) |
| lg | 24px | Section padding, form section breaks (`space-y-6`) |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- Rule row min-height: 48px touch target — inherited from `RuleRow` (`min-h-[48px]`)
- Form container max width: `max-w-2xl` with `px-4 py-8 sm:px-8` — inherited from `RuleForm` outer `<main>`
- Field label-to-select gap in `MealTemplateFields`: `space-y-2` (8px) — on-grid value; use this for all new fields in Phase 10
- Note: `SchedulingRuleFields` internally uses a 6px label-to-select gap (pre-existing, out of Phase 10 scope). `MealTemplateFields` does NOT inherit this non-grid value; it uses `space-y-2` (8px) instead.

## Color

All colors from shadcn CSS variables (olive theme). Do not hardcode hex values; use CSS variable references via Tailwind utility classes.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` (oklch 1 0 0 = white) | Page background, form surface |
| Secondary (30%) | `--muted` / `--card` | Fieldset areas, chip group container background |
| Accent (10%) | `--primary` (oklch 0.852 0.199 91.936 = golden-olive) | Active toggle switch, Save Rule button, active TabsTrigger, active Toggle chip for allowed_slots |
| Destructive | `--destructive` (oklch 0.577 0.245 27.325 = red) | Delete confirmation alert, Delete button variant |

Accent reserved for:
- Save Rule button (primary Button variant) — the only primary CTA in the form
- Rule enabled/disabled toggle switch (`bg-primary` when enabled)
- Active `TabsTrigger` selected state (shadcn default behaviour)
- Active allowed_slots chip (Toggle pressed state — use `data-[state=on]:bg-primary data-[state=on]:text-primary-foreground`)

Muted foreground (`--muted-foreground`): impact preview descriptions, hint text below context scope section, field helper text, muted rule summaries.

Warning color: `border-amber-500 bg-amber-50 text-amber-900` — impact preview zero-match alert. Keep exact class strings; do not replace with design tokens.

### Modified: `RuleImpactPreview.tsx`

Add `meal-template` case:

When `formState.ruleType === 'meal-template'` and `formState.base_type !== ''`:

Count all components in `allComponents` where `componentType === 'base'` (bases of the selected `base_type` are not directly filterable from the component model — base_type is a property on meals, not components; instead, count all base components as the affected pool).

Display: `"This template applies to {N} base components of type {base_type}."` where N = count of `allComponents.filter(c => c.componentType === 'base').length`.

If `base_type === ''`: return null (no preview).

No zero-match alert for meal-template — the impact is structural, not a count that can hit zero in normal usage.

## Form Validation Contract

Save Rule button (`Button type="submit"`) is **disabled** until all of:

1. `name.trim() !== ''`
2. `ruleType === 'meal-template'`
3. `base_type !== ''` (required)
4. At least one constraint is set (per CONTEXT.md D-05):
   - `allowed_slots.length > 0`, OR
   - `exclude_component_types.length > 0`, OR
   - `exclude_extra_categories.length > 0`, OR
   - `require_extra_category !== null`

A meal-template with only `base_type` and zero constraints is a no-op and must not be saveable.

Days and slots (context scope fields) are optional — no minimum required.

## Impact Preview Contract

`RuleImpactPreview` case for `meal-template`:

- Show when `base_type !== ''`
- Display: `"This template applies to {N} base components."` where N = count of all base components in the library
- No amber warning — structural template, not a pool filter
- No preview when `base_type === ''`

## Interaction States

### Rule type tab set (after Phase 10)

Three tabs: `no-repeat` | `scheduling-rule` | `meal-template`

Before any tab selected: Save button disabled. Switching tabs preserves `name` field value, resets type-specific fields.

### Meal Template tab — base type selection

RadioGroup: default unselected. Selecting any option sets `base_type`. Form remains invalid until at least one constraint is also set.

### Allowed slots chip group

Default: all chips unselected (= unrestricted, allowed_slots = []). Each chip toggles independently. There is no "all slots" meta-chip — empty selection is the unrestricted state.

Selecting one or more chips populates `allowed_slots` and satisfies the constraint requirement.

### Exclude component types checkboxes

Two checkboxes: Curry, Subzi. Independent toggles. Each checked item is added to `exclude_component_types[]`. Both unchecked = empty array.

### Exclude extra categories checkboxes

Five checkboxes: Liquid, Crunchy, Condiment, Dairy, Sweet. Independent toggles. Each checked item is added to `exclude_extra_categories[]`.

### Require extra select

Default: "None (optional)" (value `''`). Selecting a category sets `require_extra_category`. Re-selecting "None (optional)" clears it back to `null`.

### Composition scope — Days

Checkbox group pattern identical to `SchedulingRuleFields` Section 4. Unchecked all = null (applies to all days). No minimum required.

### Composition scope — Slots

Checkbox group pattern identical to `SchedulingRuleFields` Section 5. Unchecked all = null (applies to all slots). No minimum required.

### Rule row toggle (existing)

Custom toggle switch (inline `<button role="switch">`) — unchanged from existing `RuleRow`. Migrated meal-template rules toggle and delete identically to user-created rules.

### Delete confirmation (existing)

Inline expand below the rule row. Alert variant destructive. Two buttons: Delete (destructive) + Keep rule (ghost). Unchanged from existing `RuleRow`.

### Migration (silent)

Migration from Dexie v6 to v7 is **silent** — no banner, no notification, no UI feedback. Migrated rules appear in the rules list immediately after upgrade runs at app startup. `ruleDescriptions.ts` and `RuleRow.ruleTypeLabel` already handle `meal-template` and `scheduling-rule` (migrated component_slot_overrides) descriptions — no rendering changes needed.
